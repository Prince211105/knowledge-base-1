# Common misconceptions about the WBS

## Overview

Explains typical misunderstandings about the WBS and corrects them.

## Detailed Summary

Project teams often incorrectly describe the WBS as a schedule, a list of time estimates, a work-assignment tracking tool, or a deliverable created solely to estimate time and costs. These views misunderstand both the purpose and content of the WBS. The WBS is not a scheduling or budgeting deliverable and does not contain activities, cost estimates, resource assignments, or activity dependency information.

## Content

Typical misconceptions about the WBS include believing that it is: a schedule created at the beginning of the project, a document showing time estimates for project activities, a tracking deliverable for team members' work assignments, or a required input only for generating time and cost estimates. These interpretations are incorrect because the WBS is deliverables-based. It does not contain activities, cost estimates, resource assignments, or dependency information. Instead, it supports the creation of schedules and budgets by clarifying what outputs the project must produce.

## Best Practices

- Teach stakeholders the difference between deliverables and activities before decomposition begins.
- Review proposed WBS elements and remove any that describe time, cost, or sequencing.

## Examples

- A team that creates only a task list with dates has made a schedule, not a WBS.

## Warnings

- Treating the WBS as a schedule can cause scope gaps and planning confusion.

## Limitations

- Correcting misconceptions alone does not create a complete WBS; decomposition and validation are still required.

## Troubleshooting

- If stakeholders propose milestones, dependencies, or estimates in the WBS, move those items to the project schedule or budget artifacts.

## Decision Rules

- If an item expresses dates, durations, costs, dependencies, or assignments, it does not belong in the WBS.
