# WBS versus Project Schedule

## Overview

Clarifies the difference between a deliverables-based WBS and an activities-based Project Schedule.

## Detailed Summary

The WBS is not the Project Schedule. The WBS contains deliverables and deliverable hierarchy only, while the Project Schedule contains activities, tasks, milestones, timing dependencies, durations, and resource assignments. Every WBS deliverable is eventually decomposed into activities to produce it. Activities are the "How's" for producing project deliverables, while the WBS captures the "What's." The schedule is based on the WBS.

## Content

The WBS and Project Schedule are related but distinct artifacts. The WBS defines what the project will produce. The Project Schedule defines how and when the project team will produce it. Scheduling information includes activities, milestones, sequence logic, durations, and resources. None of that belongs in the WBS. The WBS is the blueprint that justifies the schedule. As planning progresses, each deliverable on the WBS is translated into activities, and those activities are then sequenced, resourced, and timed in the schedule.

## Best Practices

- Complete deliverable decomposition before detailed activity planning.
- Use the WBS as the scope baseline for schedule development.

## Examples

- Deliverable: Course Location Rooms is a WBS item. Booking rooms and setting up rooms are schedule activities.

## Warnings

- If the WBS contains tasks or milestones, stakeholders may confuse scope definition with execution planning.

## Limitations

- The WBS alone cannot represent the dynamic timing and dependency logic of the schedule.

## Troubleshooting

- If planning artifacts are mixed, separate nouns into the WBS and verbs into the schedule.

## Decision Rules

- If the element answers "what will be produced," place it in the WBS.
- If it answers "how, when, or by whom," place it in the schedule or related planning artifact.
