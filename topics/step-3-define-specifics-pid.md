# Define In‑Scope, Out‑Of‑Scope, Requirements and Deliverables in a Project Initiation Document

Specifics answer what exactly must be delivered:
- List of in‑scope items and explicit out‑of‑scope items.
- Initial requirements already defined (functional, non‑functional).
- Project boundaries (what the team must not exceed).
- Deliverables with acceptance criteria, assumptions, and constraints (e.g., number of revisions permitted).
Providing clear specifics reduces ambiguity and protects against later disputes.

## Can Answer

How can I specify in‑scope, out‑of‑scope, requirements and deliverables in a Project Initiation Document?

## Examples

- In‑scope: Design and develop user registration flow.
- Out‑of‑scope: Integration with third‑party payment gateway (planned for Phase 2).
- Assumption: Client will provide branding assets within 5 business days.

## Warnings

- Leaving out‑of‑scope items undefined can lead to scope creep.
