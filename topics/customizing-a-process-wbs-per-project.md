# Customizing a process WBS per project

## Overview

Explains how a reusable process WBS is tailored to each project's scope.

## Detailed Summary

A process WBS is ideally a blueprint that includes every type of deliverable that could be part of performing the process. For each project where the process is used, the project's scope determines which process deliverables are in scope and which are out of scope. This reduces the likelihood of overlooking a process deliverable and encourages consistent execution across projects.

## Content

A reusable process WBS should be broad enough to cover all deliverable types relevant to the methodology. Individual projects then customize that blueprint by keeping only the deliverables that apply to their scope. This approach supports consistency while allowing flexibility. It also reduces the risk that a project team will inadvertently miss a process deliverable because the standard blueprint already makes it visible.

## Best Practices

- Maintain a broad process WBS blueprint.
- Tailor it per project by removing out-of-scope deliverables.

## Examples

- A workplace relocation blueprint may contain all possible relocation deliverables, while a specific relocation project includes only those actually needed.

## Warnings

- Using a generic process WBS without tailoring can include unnecessary scope.

## Limitations

- A reusable blueprint still requires project-specific review and decomposition.

## Troubleshooting

- If teams are overlooking process steps, verify whether they are using and tailoring the standard process WBS appropriately.

## Decision Rules

- Treat the reusable process WBS as a starting point, not an automatic final scope for every project.
