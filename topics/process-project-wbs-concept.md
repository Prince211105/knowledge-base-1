# Process project WBS concept

## Overview

Describes how a WBS applies to projects that define a process or methodology.

## Detailed Summary

A project that defines a process will have a WBS documenting all components of the process. Such process deliverables may be used repeatedly in ongoing operations or to execute specific types of projects, such as a software development methodology or an office relocation process. A process WBS can function as a blueprint that includes every type of deliverable that could be part of performing the process.

## Content

Process projects produce repeatable methods rather than a traditional physical product. Their WBS captures the deliverables that make up the process or methodology so the organization can use the process consistently. The process WBS should ideally include every kind of deliverable that could be part of executing the process, after which each individual project can tailor the scope by selecting which process deliverables are in or out of scope.

- How processes are represented in a WBS

- How process scope is tailored per project

## Best Practices

- Build a reusable process WBS that includes all potentially relevant deliverables.
- Tailor the process WBS per project to reflect actual scope.

## Examples

- A workplace relocation WBS can document deliverables used each time a relocation process is executed.

## Warnings

- If a process WBS is not tailored, teams may plan unnecessary deliverables or overlook needed ones.

## Limitations

- A generic process WBS may not be fully decomposed for every project context.

## Troubleshooting

- If process steps are inconsistent between projects, review whether the reusable WBS blueprint is complete and being tailored correctly.

## Decision Rules

- Use a reusable process WBS as a starting blueprint, then remove out-of-scope deliverables for each project.
