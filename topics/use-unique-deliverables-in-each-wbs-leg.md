# Use unique deliverables in each WBS leg

## Overview

Explains the guideline that each WBS branch should contain unique deliverables.

## Detailed Summary

An effective WBS ensures each leg contains deliverables that are unique from deliverables in another leg. The same deliverable should not be repeated in each column or branch. This helps maintain clarity, avoid duplication, and preserve logical relationships in the scope structure.

## Content

Each WBS branch should represent a distinct portion of the project scope. Repeating the same deliverable across multiple branches can confuse ownership, distort completeness checks, and weaken the usefulness of the structure. Unique placement of deliverables helps stakeholders understand how the scope is organized and prevents accidental double representation of the same output.

## Can Answer

- Why duplicated deliverables are a problem
- How to structure unique WBS branches

## Best Practices

- Assign each deliverable to the single most logical branch.
- Review the WBS for repeated items during validation.

## Examples

- Instead of placing the same approval document under several branches, place it once in the most appropriate deliverable leg.

## Warnings

- Duplicate deliverables can lead to confusion and perceived overstatement of scope.

## Limitations

- Some supporting artifacts may relate to multiple areas, but the WBS should still place them uniquely.

## Troubleshooting

- If a deliverable seems to fit multiple branches, select the branch that best reflects its primary scope relationship.

## Decision Rules

- Do not repeat the same deliverable in multiple WBS legs; place it once in the most logical location.
